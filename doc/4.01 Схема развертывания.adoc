=== Схема развертывания

==== География и ЦОД

* Москва — северный регион, основной ЦОД
* Краснодар — южный регион, DR/частично active-active
* Между ЦОД — асинхронная репликация критичных данных
* CDN для медиа — edge: СПб и Казань

---

==== Сервисы и распределение по ЦОД

===== API и бизнес-сервисы

* API Gateway — контейнеры
- Москва: 4 реплики
- Краснодар: 2 реплики
- Stateless, active-active

* Сервис авторизации — контейнеры + базы
** PostgreSQL (users, sessions)
-  Москва: Master shard 1
- Краснодар: Master shard 2
- Реплики shard 1 в Краснодар, shard 2 в Москву
* Redis cluster (sessions + tokens)
- Москва: Master 3 + 1 replica
- Краснодар: Master 3 + 1 replica
- Cluster mode, hash-slot шардирование
- Все записи пользователей и сессий идут в локальный мастер ЦОД

* Сервис пользователей — контейнеры + PostgreSQL
- Москва: Master shard 1
- Краснодар: Master shard 2
- Реплики shard 1 в Краснодар, shard 2 в Москву
- Обеспечивает локальную запись пользователей и отказоустойчивость при падении одного ЦОД

* Сервис объявлений — VM + PostgreSQL
- Москва: Master shards 1,2; Replica shard 3
- Краснодар: Master shard 3; Replica shards 1,2
- Geo-sharding: запись в локальный мастер, реплика в другой ЦОД
- Шардирование обеспечивает балансировку нагрузки и отказоустойчивость

* Сервис поиска + рекомендации — VM + ElasticSearch
- Москва: 3 data node, 1 replica shard
- Краснодар: replica shard
- Каждый shard имеет одну копию на другой ноде для отказоустойчивости

* Сервис заказов, платежей, расчётов — VM + PostgreSQL
- Master shard по региону с репликой в другом ЦОД (рекомендуется для нагрузки)
- Вариант single-master + DR при строгой консистентности

* Сервис доставки — VM + PostgreSQL
- Geo-sharding по региону
- Master в локальном ЦОД, replica в другом

* Аналитический сервис — VM + ClickHouse
- Москва: 2 ноды
- Краснодар: 1 нода
- Обеспечивает аналитическую нагрузку и уведомления

* Сервис медиа — S3 + CDN
- S3 Москва (primary)
- S3 Краснодар (replica)
- CDN edge СПб и Казань для быстрого доступа

===== Технические компоненты

* Redis cluster — кэш для сессий, счетчиков
- Москва: 3 master + 1 replica
- Краснодар: 3 master + 1 replica
- Cluster mode, hash-slot шардирование

* Kafka — event bus
- Москва: 2 брокера
- Краснодар: 1 брокер
- replication factor = 3

* Мониторинг — ELK stack
- Москва: 1 stack
- Краснодар: 1 stack
- Централизованное логирование

* Метрики — Prometheus + Grafana
- Москва: 1 инстанс
- Краснодар: 1 инстанс
- Сбор ключевых метрик и alerting

---

==== Принцип работы

* Stateless-сервисы → контейнеры в Kubernetes, Helm, active-active по ЦОД
* Stateful-сервисы → master в локальном ЦОД + реплика в другом
* Geo-sharding → запись в локальный мастер shard, репликация в другой ЦОД
* Failover → реплики становятся мастерами, Kubernetes масштабирует контейнеры, GeoDNS переключает трафик
* Нагрузка → реплики рассчитаны на пик ~4500 rps, всплески ~9000 rps, запас N+1
