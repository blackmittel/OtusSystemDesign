==== География и ЦОД

* Москва — северный регион, основной ЦОД (primary для критичных записей)
* Краснодар — южный регион, secondary / standby (частично active-active для некритичных сервисов)
* Казань — witness-узел (узел кворума для отказоустойчивости)

* Между ЦОД:
- асинхронная репликация данных (по умолчанию)
- синхронная репликация внутри ЦОД для критичных сервисов

* CDN для медиа:
- edge: СПб и Казань

==== Сервисы и распределение по ЦОД

===== API и бизнес-сервисы

* API Gateway — контейнеры
- Москва: 4 реплики
- Краснодар: 2 реплики
- Stateless, active-active

* Сервис авторизации — контейнеры + базы (VM/bare-metal)
** PostgreSQL (users, sessions)
-  Москва: Master shard 1
- Краснодар: Master shard 2
- Реплики shard 1 в Краснодар, shard 2 в Москву
- Локальная запись (по географии)
* Redis cluster (sessions + tokens)
- Москва: 3 master + 3 replica
- Краснодар: 3 master + 3 replica
- Cluster mode, hash-slot
- Все записи пользователей и сессий идут в локальный мастер ЦОД
- Асинхронная меж-ЦОД репликация

---

* Сервис пользователей — контейнеры + PostgreSQL (VM/bare-metal)
- Москва: Master shard 1
- Краснодар: Master shard 2
- Реплики shard 1 в Краснодар, shard 2 в Москву
- Обеспечивает локальную запись пользователей и отказоустойчивость при падении одного ЦОД

* Сервис объявлений — контейнеры + PostgreSQL (VM/bare-metal)
- Москва: Master shards 1,2; Replica shard 3
- Краснодар: Master shard 3; Replica shards 1,2
- Geo-sharding: запись в локальный мастер, реплика в другой ЦОД
- Шардирование обеспечивает балансировку нагрузки и отказоустойчивость

* Сервис поиска + рекомендации — контейнеры + ElasticSearch (VM/bare-metal)
- Москва: 3 data node
- Краснодар: 3 data node
- replication factor = 1 (меж-ЦОД)
- eventual consistency

* Сервис заказов, платежей, расчётов — контейнеры + PostgreSQL (VM/bare-metal)
- Модель: Single-writer (единственный primary)

- Развёртывание:

*** Москва:
****  - 1 primary
****  - 1 synchronous replica
*** Краснодар:
****  - 1 asynchronous replica (standby)

- Failover:
** - автоматический, при кворум (обеспецивается etcd, см. ниже)
** - при недоступности primary временная остановка записи до принятия решения

---

* Сервис доставки — контейнеры + PostgreSQL (VM/bare-metal) (критичный контур)
- Geo-sharding по региону
- Master в локальном ЦОД, replica в другом

* Аналитический сервис — контейнеры + ClickHouse (VM/bare-metal) 
- Москва: 2 ноды
- Краснодар: 2 ноды
- Асинхронная репликация

---

* Сервис медиа — S3 + CDN
- S3 Москва (primary)
- S3 Краснодар (replica)
- CDN edge: СПб, Казань

===== Технические компоненты

* Redis cluster (cache)
- Москва: 3 master + 3 replica
- Краснодар: 3 master + 3 replica
- Асинхронная репликация

---

* Kafka — event bus
- Москва: 3 брокера
- Краснодар: 3 брокера
- replication factor = 3
- Меж-ЦОД распределение партиций

---

* etcd cluster (кворум при падениях)

- Назначение:

** - определение "жив ли primary", предотвращение двух primary для критичных сервисов
** - защита от split-brain
** - координация failover PostgreSQL (через Patroni)

- Состав:
** - Москва: 2 ноды
** - Краснодар: 2 ноды
** - Казань: 1 witness node
(резервные ноды, кворум = 3/5)

---

* Мониторинг — ELK stack
- Москва: 1 кластер
- Краснодар: 1 кластер
- Централизованное логирование

---

* Метрики и health-checks — Prometheus + Grafana

- Развёртывание:
** - Москва: 2 инстанса Prometheus
** - Краснодар: 2 инстанса Prometheus

* Дополнительно:
- health agents
** - проверка:
*** - доступности PostgreSQL primary
*** - доступности ЦОД (network checks)

** -Использование:
*** - сигнализация отказов
*** - триггеры для failover (через интеграцию с etcd/Patroni)

==== Принципы

* Stateless-сервисы:
- Kubernetes, active-active

* Stateful-сервисы (БД etc.):
- локальный primary + реплика в другом ЦОД

* Критичные сервисы (Payments, Orders):
- single-writer модель
- failover только через кворум etcd
- при неопределённости запись останавливается

* Geo-sharding:
- локальная запись для пользователей и объявлений (Ticket)

* Failover:
- автоматический, но с проверкой quorum
- Kubernetes - пересоздание контейнеров
- PostgreSQL - переключение на реплику только при подтверждении etcd
- GeoDNS - переключение трафика
