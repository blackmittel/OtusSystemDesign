=== CI/CD процесс

==== Инструменты
* Git: GitVerse
* CI/CD: GitVerse CI/CD
* Контейнеризация: Docker
* Оркестрация: Kubernetes
* Infrastructure as Code: OpenTofu
* Упаковка приложений: Helm
* Registry: GitVerse Container Registry
* Мониторинг и alerting: Prometheus + Grafana
* Логирование: ELK Stack
* Миграции баз данных: Liquibase, Flyway
* Секреты: Vault

==== Архитектура пайплайнов
Система использует **три отдельных пайплайна**, разделение по назначению и чувствительности данных:

===== **Пайплайн бизнес-сервисов**
- Сервисы: API Gateway, Авторизация, Пользователи, Объявления, Поиск, Заказы, Платёжный сервис, Расчёты, Доставка, Аналитика/Нотификации
- Контейнеризация: Docker + Kubernetes
- Стратегия: Blue-Green развертывание, Canary для критичных сервисов (Заказы, Платёж, Расчёты)


===== **Пайплайн баз данных и инфраструктуры**
- Сервисы: PostgreSQL, Redis, Kafka, ClickHouse, Elasticsearch
- Развертывание: VM / bare-metal
- Стратегия: контроль версий через OpenTofu, репликация между ЦОД, автоматическое тестирование восстановления
- Миграции БД: только через Flyway, обязательный бэкап перед применением, ручное подтверждение для критичных

===== **Пайплайн медиа и CDN**
- Сервисы: S3/Object Storage, CDN edge (СПб, Казань)
- Развертывание: managed
- Стратегия: автоматическое обновление правил кэширования и деплой новых версий контейнеров медиа-сервисов

==== Этапы CI/CD

===== CI

* Trigger: push/merge request в GitVerse
* Этапы:
1. Сборка Docker-образа
2. Unit-тесты (>80% покрытия)
3. Интеграционные тесты для взаимодействия сервисов
4. Статический анализ (SonarQube)
5. Security scan (Trivy, Grype)
6. Сохранение артефактов в GitVerse Registry

===== CD

* Стратегия: Canary для критичных сервисов, Blue-Green для остальных
* Окружения:
- Development — автоматический деплой feature-веток
- Preproduction — полная копия production, интеграционное тестирование
- Production — деплой по расписанию, ночное время 02:00-04:00
* Шаги:
1. Создание Helm-чарта с новой версией
2. Развертывание в preproduction
3. Запуск интеграционных и e2e тестов
4. Деплой в production в соответствии с моделью для сервиса
5. Мониторинг задержек, error rate, RPS
6. Автоматический rollback при превышении порогов

===== Мониторинг и уведомления
* Health checks для всех сервисов после деплоя
* Метрики в Prometheus/Grafana
* Уведомления администраторов при критических событиях
* Централизованное логирование через ELK
