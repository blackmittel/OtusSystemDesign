== 5. Система хранения данных

=== 5.1 Общая архитектура

Система хранения данных разделена по бизнес-доменам и типам нагрузки. Используется комбинация реляционных и нереляционных хранилищ, кэширования и асинхронной обработки для масштабируемости и отказоустойчивости.

[cols="1,1,1,2,3",options="header"]
|===
| Компонент | Тип хранилища | Геораспределение | Репликация / Шардинг | Особенности

| Сервис пользователей
| PostgreSQL
| Москва / Краснодар (master-slave)
| Шардирование по ID пользователя, реплика для чтения в каждом ЦОД
| ACID-транзакции, профили, списки, безопасное хранение хешированных паролей

| Сервис объявлений
| PostgreSQL + CQRS (write: PostgreSQL, read: ElasticSearch)
| Москва / Краснодар (master-slave) + ES cluster с geo-shards
| Шардирование по ID объявления; ES индекс с репликами в ЦОД
| Write через PostgreSQL, read через ES; CQRS позволяет масштабировать поиск и рекомендации

| Сервис поиска / рекомендации
| ElasticSearch
| Москва / Краснодар + региональные копии
| Реплики для чтения; индексация событий через Kafka
| Индекс объявлений, профилей, сообщений; отдельный кластер для рекомендаций

| Сервис заказов
| PostgreSQL
| Москва / Краснодар (master-master для отказоустойчивости)
| Шардирование по ID заказа; реплики для чтения в каждом ЦОД
| ACID-транзакции, статус заказов, связь с платежами и доставкой

| Платёжный сервис
| PostgreSQL / внешние платёжные API
| Москва
| Реплика для чтения; шифрование чувствительных данных
| Только авторизованные транзакции; интеграция с Kafka для событий платежа

| Сервис расчётов
| PostgreSQL
| Москва
| Шардирование по ID пользователя / продавца; реплика для аналитики
| Выплаты, баланс; события через Kafka

| Сервис доставки
| PostgreSQL
| Москва / Краснодар
| Шардирование по ID заказа; реплики для чтения
| Расчёт стоимости, статус доставки; интеграция с Kafka

| Сервис медиа
| S3 / совместимый объектный сторедж
| CDN (СПб, Казань) + Москва/Краснодар (origin)
| Версионирование объектов, региональные кэш-копии
| Хранение изображений, видео; edge-caching через CDN

| Кэш
| Redis Cluster
| Москва / Краснодар
| Мастер-слейв + репликация; TTL для токенов и временных данных
| Токены авторизации, сессии, кэширование ответов сервисов

| Асинхронные события
| Kafka
| Москва / Краснодар (multi-broker cluster)
| Репликация топиков по ЦОД; partitioning по ID сущности
| Публикация событий платежей, заказов, объявлений, сообщений, поведения пользователя

| Мониторинг / логирование
| ELK (Elasticsearch / Logstash / Kibana)
| Москва
| Репликация логов, хранение в отдельных индексах по сервисам
| Централизованный сбор логов, корреляция событий, анализ ошибок
|===

=== 5.2 Рекомендации по репликации и шардированию

* **PostgreSQL**

* Master-slave репликация между ЦОДами для отказоустойчивости.
* Геошардинг по ID пользователя или ID объявления для равномерной нагрузки.
* Read-replicas для масштабирования чтения (поиск профиля, истории заказов).
* Использовать logical replication для аналитики и CQRS-подсистем (ElasticSearch).

* **ElasticSearch**

* Разделение индексов на шардов и реплик.
* Геораспределение: основной кластер в Москве, реплики в Краснодаре.
* Hot/warm архитектура для свежих и архивных данных.
* Асинхронная индексация через Kafka для минимизации нагрузки на сервис объявлений.

* **Redis**

* Кластеризация с мастерами и репликами для каждого ЦОД.
* TTL для токенов и кэшированных данных.
* Использовать partitioning (hash slots) для распределения нагрузки.

* **S3 / объектное хранилище**

* Версионирование и региональные копии.
* CDN для медиаконтента, edge-кеширование в СПб и Казани.
* Асинхронная обработка (например, генерация миниатюр) через Kafka или worker queue.

* **Kafka**

* Разделение топиков по типу события и сущности (orders, payments, ads, messages).
* Гео-репликация между ЦОДами.
* Использование compacted topics для ключевых сущностей (например, баланс пользователей).

=== 5.3 CQRS и Event Sourcing

* Сервис объявлений:

* Write модель: PostgreSQL для транзакционных операций (создание, редактирование, удаление).
* Read модель: ElasticSearch для поиска и рекомендаций.
* Все изменения публикуются в Kafka для асинхронной индексации и аналитики.
* Event Sourcing:

* Опционально можно использовать для заказов и расчётов — хранение всех событий (создание заказа, оплата, доставка) для полной истории и восстановления состояния.
