=== Система хранения данных

==== Общая архитектура

Система хранения данных разделена по бизнес-доменам и типам нагрузки. Используется комбинация реляционных и нереляционных хранилищ, кэширования и асинхронной обработки для масштабируемости и отказоустойчивости.

==== Репликация и шардирование

* **PostgreSQL**

* Geo-sharding по user_id / ticket_id / order_id.
* Master-Master для Users, Tickets, Orders, Delivery.
* Single-writer (Primary + replicas) для Payments и Billing.
* Read-replicas для масштабирования чтения.
* Logical replication для интеграции с Kafka / аналитикой.

* **ElasticSearch**

* Индексы разделены на shards и replicas.
* Кластеры в каждом ЦОД.
* Синхронизация через Kafka.
* Поддержка full-text поиска и фильтрации.

* **Redis**

* Кластеры в каждом ЦОД.
* Master + replicas.
* TTL для токенов и кэша.
* Партиции (hash slots).

* **S3 / объектное хранилище**

* Репликация между регионами.
* CDN (СПб, Казань) для edge-кеширования.
* Асинхронная обработка медиа через очереди.

* **Kafka**

* Топики по доменам (orders, payments, tickets, users).
* Партиции по ключу (entity_id).
* Репликация между ЦОД.
* Использование compacted topics для состояний.

==== CQRS и Event-driven

*Сервис Tickets:*

* Write модель: PostgreSQL.
* Read модель: ElasticSearch.
* Изменения публикуются в Kafka.

*Event-driven архитектура:*

* Все сервисы публикуют события в Kafka.
* Подписчики обрабатывают события асинхронно.
* Используется для индексации, аналитики, рекомендаций
