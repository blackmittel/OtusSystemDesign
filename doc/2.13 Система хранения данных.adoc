=== Система хранения данных

==== Общая архитектура

Система хранения данных разделена по бизнес-доменам и типам нагрузки. Используется комбинация реляционных и нереляционных хранилищ, кэширования и аналитического хранилища для масштабируемости и разделения OLTP/OLAP нагрузки.

==== Репликация и шардирование

* PostgreSQL
  - geo-sharding по user_id / ticket_id / order_id
  - master-replica или single-writer по домену
  - read-replicas для масштабирования

* Elasticsearch
  - shard + replica модель
  - синхронизация через Kafka

* Redis
  - cluster mode
  - TTL для сессий и токенов

* S3 / CDN
  - репликация между регионами
  - edge-кэширование (СПб, Казань)

* Kafka
  - partitioning по entity_id
  - replication между ЦОД

* ClickHouse (аналитика)
  - ReplicatedMergeTree
  - Distributed tables между Москвой и Краснодаром
  - ingestion через Kafka Engine
  - materialized views для агрегаций

==== CQRS и Event-driven

* Сервис объявлений:
  - write: PostgreSQL
  - read: Elasticsearch
  - события публикуются в Kafka

* Event-driven слой:
  - все сервисы публикуют события в Kafka
  - ClickHouse потребляет события для аналитики
  - аналитический сервис формирует агрегаты и метрики

etcd-кластер из трёх узлов:
- Москва: leader
- Краснодар: follower
- Казань: witness (участвует в кворуме, не хранит данные)

При отказе Москвы кворум (Краснодар + Казань) выбирает новый мастер для платежей и расчётов.
