=== Планы тестирования
==== Назначение

Цель тестирования — подтвердить

* корректную работу при распределении по ЦОДам
* поддержку средних и пиковых нагрузок
* сохранность данных при сбоях и задержках
* непрерывность ключевых бизнес-процессов (объявления, заказы, оплаты)
* корректность OLTP + OLAP разделения
* отсутствие потерь данных (включая события)
* соответствие требованиям по доступности и восстановлению

==== Объекты тестирования

===== Точки входа

* GeoDNS (маршрутизация пользователей между регионами)
* API Gateway (единая точка входа)
* CDN (доставка медиа-контента)

===== Бизнес-сервисы

* авторизация и управление пользователями
* сервис объявлений
* поиск и рекомендации
* заказы
* платежи и расчёты
* доставка
* уведомления
* аналитика (OLAP-запросы)

===== Инфраструктурные компоненты

* базы данных (PostgreSQL)
* кэш (Redis)
* очередь сообщений (Kafka)
* поисковый индекс (ElasticSearch)
* аналитическое хранилище OLAP (ClickHouse)
* хранилище медиа (S3 + CDN)
* системы логирования и мониторинга (ELK, Prometheus, Grafana)

==== Архитектурные особенности

При тестировании учитываются следующие свойства системы:

* два активных дата-центра (Москва и Краснодар)
* распределённое хранение данных с разделением по регионам
* асинхронное взаимодействие сервисов через события
* разделение OLTP (PostgreSQL) и OLAP (ClickHouse)
* возможная задержка согласования данных между системами
* replay событий как механизм восстановления

==== Основные группы тестирования

===== Функциональная проверка системы

Проверяется, что бизнес-сценарии работают от начала до конца:

* регистрация и вход пользователя
* создание и публикация объявления
* поиск и просмотр объявлений
* оформление заказа
* оплата и подтверждение
* доставка и завершение заказа

Дополнительно:
* корректность формирования событий в Kafka
* попадание событий в ClickHouse

---

===== Проверка согласованности данных

Проверяется:

* PostgreSQL — источник истины
* Elasticsearch — синхронизация индекса ≤ 2 мин
* Redis — TTL consistency ≤ 60 сек
* ClickHouse:
  - задержка событий ≤ 60–180 сек
  - отсутствие потерь событий (через replay)
* допустимая eventual consistency:
  - OLTP vs OLAP расхождение ≤ 5 минут

---

===== Проверка взаимодействия сервисов (event-driven)

Проверяется:

* доставка событий Kafka:
  - success rate ≥ 99.99%
* повторная обработка (idempotency)
* обработка out-of-order событий
* работа consumer groups при масштабировании
* replay из Kafka в ClickHouse

---

===== Проверка распределённой системы (2 ЦОД)

Проверяется:

* переключение GeoDNS ≤ 60 сек
* отказ одного ЦОД:
  - деградация ≤ 30% по latency
* восстановление:
  - ≤ 15 минут частичное восстановление
* отсутствие потерь заказов и платежей

---

===== Проверка производительности

Проверяется способность системы работать при разных уровнях нагрузки:

* normal load:
  - 900 req/sec
* peak load:
  - 4500 req/sec
* burst load:
  - 9000 req/sec

Проверяются:

* Kafka, пропускная способность:
  - 50k–100k events/sec
* ClickHouse, задержка загрузки данных:
  - ≤ 60 сек задержка при норме
* ES отклик поиска:
  - ≤ 200 ms P95

---

===== Проверка устойчивости к сбоям

Сценарии:

* отказ PostgreSQL shard
* отказ Redis cluster node
* остановка Kafka broker
* деградация ClickHouse node
* отказ одного ЦОД
* потеря связи между ЦОД

Критерии:

* деградация без падения системы
* восстановление ≤ 15 минут (services)
* восстановление данных через replay ≤ 6 часов

---

===== Проверка восстановления

Проверяется:

* восстановление PostgreSQL из реплики
* replay Kafka => восстановление ClickHouse
* восстановление ES индекса
* восстановление Redis cache

---

===== Проверка безопасности

* TLS 1.2+
* AES-256 на диске
* ограничение запросов:
  - ≤ 100 req/sec per user
* защита от SQL/XSS
* проверка подписи webhook'ов

---

==== Критические зоны системы

* платежи и финансовые операции
* создание и изменение заказов
* публикация объявлений
* поиск и выдача результатов
* авторизация и управление сессиями
* передача событий (Kafka → ClickHouse)

---

==== Общий критерий успешности тестирования

Система считается корректной, если:

* 100% критических сценариев проходят успешно
* потери транзакций: 0
* потери событий: 0 (с учетом replay)
* соблюдение SLA:
  - latency ≤ 2 сек (P95)
  - search ≤ 5 сек (P95)
* система выдерживает:
  - 10k concurrent users (stable)
  - 100k events/sec ingestion
* восстановление:
  - сервисы ≤ 15 мин
  - данные ≤ 1 час (OLTP), ≤ 6 часов (OLAP)
* геораспределение не приводит к потере консистентности выше допустимого порога
