=== Планы тестирования
==== Назначение

Цель тестирования — подтвердить

* корректную работу при распределении по ЦОДам
* поддержку средних и пиковых нагрузок
* сохранность данных при сбоях и задержках
* непрерывность ключевых бизнес-процессов (объявления, заказы, оплаты)
* соответствие требованиям по доступности и восстановлению

==== Объекты тестирования

===== Точки входа

* GeoDNS (маршрутизация пользователей между регионами)
* API Gateway (единая точка входа)
* CDN (доставка медиа-контента)

===== Бизнес-сервисы

* авторизация и управление пользователями
* сервис объявлений
* поиск и рекомендации
* заказы
* платежи и расчёты
* доставка
* аналитика и уведомления

===== Инфраструктурные компоненты

* базы данных (PostgreSQL)
* кэш (Redis)
* очередь сообщений (Kafka)
* поисковый индекс (ElasticSearch)
* аналитическое хранилище (ClickHouse)
* хранилище медиа (S3 + CDN)
* системы логирования и мониторинга (ELK, Prometheus, Grafana)

==== Архитектурные особенности

При тестировании учитываются следующие свойства системы:

* два активных дата-центра (Москва и Краснодар)
* распределённое хранение данных с разделением по регионам
* асинхронное взаимодействие сервисов через события
* различие между синхронными и асинхронными процессами
* возможная задержка согласования данных между системами

==== Основные группы тестирования

===== Функциональная проверка системы

Проверяется, что бизнес-сценарии работают от начала до конца:

* регистрация и вход пользователя
* создание и публикация объявления
* поиск и просмотр объявлений
* оформление заказа
* оплата и подтверждение
* доставка и завершение заказа

Особенность: проверка выполняется через внешние интерфейсы системы (API Gateway).

===== Проверка согласованности данных

Цель — убедиться, что данные в разных системах не расходятся.

Проверяется:

* запись данных в основной базе (PostgreSQL)
* отражение изменений в поиске (ElasticSearch)
* актуальность данных в кэше (Redis)
* корректная обработка задержек между сервисами
* временные расхождения между системами допускаются, но ограничены
* данные не должны теряться или дублироваться

===== Проверка взаимодействия сервисов

Проверяется обмен данными между сервисами:

* корректность синхронных вызовов (например, заказ + оплата)
* корректность событийного обмена (через Kafka)
* обработка повторных и запоздалых сообщений
* устойчивость к временной недоступности сервисов

===== Проверка распределённой работы (два ЦОД)

Проверяется работа системы при географическом распределении:

* корректная маршрутизация пользователей через GeoDNS
* работа системы при отказе одного дата-центра
* переключение нагрузки между регионами
* отсутствие потери данных при региональных сбоях

===== Проверка производительности

Проверяется способность системы работать при разных уровнях нагрузки:

* обычная нагрузка (ежедневная работа)
* высокая нагрузка (пиковые часы)
* резкие всплески активности

Проверяются:

* время ответа системы
* стабильность под нагрузкой
* отсутствие деградации ключевых функций

===== Проверка устойчивости к сбоям

Проверяется поведение системы при отказах:

* отключение отдельных сервисов
* недоступность базы данных или кэша
* сбой очередей сообщений
* отключение головного дата-центра
* потеря связи между дата-центрами

Цель — убедиться, что система продолжает работать в деградированном режиме.

===== Проверка восстановления

Проверяется:

* автоматическое восстановление сервисов
* восстановление после перезапуска компонентов
* восстановление после потери части инфраструктуры
* восстановление данных из резервных копий

===== Проверка безопасности

Проверяется защита системы:

* доступ только авторизованных пользователей
* защита от подделки запросов
* защита от типовых веб-атак
* ограничение частоты запросов
* защита данных при передаче и хранении

==== Критические зоны системы

Наибольшее внимание уделяется следующим процессам:

* платежи и финансовые операции
* создание и изменение заказов
* публикация объявлений
* поиск и выдача результатов
* авторизация и управление сессиями

==== Общий критерий успешности тестирования

Система считается корректной, если:

* все ключевые сценарии выполняются без ошибок
* данные сохраняются и не теряются
* система выдерживает заданную нагрузку
* сбои не приводят к полной остановке системы
* восстановление происходит в допустимые сроки
* геораспределённая работа не нарушает целостность данных
