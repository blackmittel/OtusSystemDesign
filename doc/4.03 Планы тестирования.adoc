=== Планы тестирования

=== Планы тестирования торговой площадки

==== Функциональное тестирование
* *Модульное (Unit):* Тесты отдельных функций сервисов (JUnit для Java, pytest для Python). Покрытие >80% бизнес-логики каждого сервиса.
* *Интеграционное:*
** Тестирование взаимодействия сервисов через REST API (Postman, REST Assured)
** Тестирование работы с очередями RabbitMQ
** Проверка корректности данных между PostgreSQL и Elasticsearch
* *E2E-сценарии:* Критические пользовательские сценарии: регистрация → поиск → заказ → оплата → доставка.

==== Нагрузочное тестирование
* *Базовая нагрузка:* 10 000 RPS на API Gateway, 95-й перцентиль ответа <200 мс.
* *Пиковые нагрузки:* Кратковременное увеличение до 30 000 RPS (распродажи).
* *Тестирование узких мест:* 
** Поисковые запросы в Elasticsearch (1000 запросов/сек)
** Одновременное резервирование товаров (конкурентные транзакции в PostgreSQL)
** Очереди RabbitMQ при массовых событиях

==== Тестирование отказоустойчивости
* *Отказ сервисов:* Поочередное отключение инстансов каждого сервиса (3 реплики → 2 реплики).
* *Отказ БД:* Переключение PostgreSQL master → replica, отказ узлов Elasticsearch.
* *Сетевые проблемы:* Задержки и обрыв связи с платежными провайдерами, службами доставки.
* *Восстановление:* Автоматическое переключение Redis Sentinel, восстановление RabbitMQ узлов.

==== Тестирование безопасности
* *OWASP Top 10:* Инъекции, аутентификация, чувствительные данные.
* *Ролевая модель:* Проверка доступа к API для разных ролей (покупатель, продавец, админ).
* *Валидация токенов:* Проверка JWT, сроков действия, отзыв токенов.
* *Данные платежей:* PCI DSS compliance (маскирование, шифрование).

==== Конфигурационное и миграционное тестирование
* *Миграции БД:* Тестирование скриптов миграции PostgreSQL (Flyway/Liquibase).
* *Конфигурация кластеров:* Проверка настройки шардирования, репликации, индексов Elasticsearch.
* *Обновления:* Rolling update сервисов, откат версий.

==== Мониторинг и observability-тестирование
* *Алертинг:* Проверка срабатывания алертов при деградации сервисов.
* *Метрики:* Корректность сбора метрик в Prometheus (latency, error rate, RPS).
* *Логирование:* Полнота логов в Elasticsearch для отладки инцидентов.
* *Трассировка:* Корректность построения трасс в Jaeger для E2E операций.

==== Тестирование интеграций
* *Платежные провайдеры:* Тестовые сценарии успешных/неуспешных платежей, callback'и.
* *Службы доставки:* Мокирование API для различных статусов доставки.
* *СМС/Email-сервисы:* Проверка отправки уведомлений.

==== Тестирование восстановления данных
* *Бэкапы:* Восстановление PostgreSQL из резервных копий.
* *Reindex Elasticsearch:* Полная переиндексация из PostgreSQL.
* *Восстановление S3:* Проверка доступности данных после сбоя.

==== *Периодичность:*
* *На каждом коммите:* Unit, интеграционные тесты
* *Ежедневно:* E2E сценарии, smoke-тесты
* *Еженедельно:* Нагрузочные тесты, security-сканирование
* *Перед релизом:* Полный цикл тестирования отказоустойчивости
* *Ежеквартально:* Chaos Engineering эксперименты

==== Контроль критичных цепочек:
Для ключевых операций настроены специальные проверки:
*   *"Пользователь может купить товар"*: Раз в 5 минут автоматический тест выполняет сценарий: поиск товара → добавление в корзину → оформление тестового заказа (без реального платежа).
*   *"Поиск работает"*: Постоянно проверяется скорость и качество поиска по тестовым запросам.
*   *"Платежный шлюз доступен"*: Мониторинг ответа платежных провайдеров.
