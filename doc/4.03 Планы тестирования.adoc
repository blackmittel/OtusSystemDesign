==== Планы тестирования

===== Назначение

Цель тестирования — подтвердить:

* корректную работу в распределённой multi-DC архитектуре
* поддержку средних и пиковых нагрузок
* устойчивость событийной обработки (Kafka + ClickHouse)
* корректность разделения OLTP (PostgreSQL) и OLAP (ClickHouse)
* отсутствие потерь данных (включая события)
* соблюдение SLA по задержкам и восстановлению

---

==== Объекты тестирования

===== Точки входа

* GeoDNS
* API Gateway
* CDN

===== Бизнес-сервисы

* User / Auth
* Ticket (объявления + сообщения)
* Order
* Payment / Billing
* Delivery
* Search / Recommendation
* Analytics (включая нотификации)

===== Инфраструктурные компоненты

* PostgreSQL
* Redis
* Kafka
* Elasticsearch
* ClickHouse
* S3 + CDN
* ELK / Prometheus / Grafana

---

==== Архитектурные особенности

Учитываются:

* 2 активных ЦОД (Москва, Краснодар)
* eventual consistency между регионами
* событийная архитектура (Kafka)
* разделение OLTP и OLAP
* задержки обработки событий (от генерации до аналитики)
* повторная обработка событий как механизм восстановления

---

==== Основные группы тестирования

===== Функциональное тестирование

Проверяется end-to-end:

* регистрация / логин
* создание объявления
* поиск и выдача
* оформление заказа
* оплата
* доставка

Дополнительно:

* корректность формирования событий в Kafka
* попадание событий в ClickHouse
* генерация нотификаций аналитическим сервисом

---

===== Проверка производительности

Проверяются задержки обработки запросов и пропускная способность системы.

Сценарии:

* загрузка страницы каталога
* выполнение поискового запроса
* создание заказа
* инициация платежа

Метрики (под нагрузкой):

* API latency:
- ≤ 2 сек (P95)
- ≤ 4 сек (P99)

* search latency:
- ≤ 5 сек (P95)

* payment initiation:
- ≤ 5 сек (P95)

Дополнительно:

* concurrent users:
- ≥ 10 000 (целевое)
- ≥ 15 000 (пиковое)

---

===== Проверка согласованности данных

Проверяется:

* PostgreSQL — источник истины
* Elasticsearch — синхронизация индекса ≤ 2 мин
* Redis — согласованность TTL ≤ 60 сек
* ClickHouse:
- задержка событий ≤ 60–180 сек
- отсутствие потерь событий (через повторную обработку)

* допустимая eventual consistency:
- расхождение OLTP и OLAP ≤ 5 минут

---

===== Проверка событийной архитектуры

Проверяется:

* идемпотентность обработки событий
* обработка дублирующихся событий
* обработка событий вне порядка
* повторная обработка событий (replay)

Дополнительно:

* доставка событий через Kafka:
- success rate ≥ 99.99%

* корректность работы групп потребителей при масштабировании

* восстановление аналитики через повторное чтение событий (Kafka -> ClickHouse)

---

===== Проверка распределённой системы (2 ЦОД)

Проверяется:

* переключение GeoDNS ≤ 60 сек

* отказ одного ЦОД:
- деградация по задержкам ≤ 30%

* восстановление:
- ≤ 15 минут частичное восстановление

* отсутствие потерь заказов и платежей

---

===== Проверка устойчивости к сбоям

Сценарии:

* отказ shard PostgreSQL
* отказ узла Redis
* остановка брокера Kafka
* деградация узла ClickHouse
* отказ одного ЦОД

Критерии:

* система продолжает работу с деградацией
* восстановление сервисов ≤ 15 минут
* восстановление данных через повторную обработку ≤ 6 часов

---

===== Проверка восстановления

Проверяется:

* восстановление PostgreSQL из реплики
* восстановление аналитики через повторную обработку Kafka -> ClickHouse
* восстановление индекса Elasticsearch
* восстановление Redis cache

---

===== Проверка безопасности

* TLS 1.2+
* шифрование данных (AES-256)
* rate limiting:
- ≤ 100 req/sec на пользователя (по умолчанию)
* защита от SQL/XSS
* проверка подписи webhook

---

==== Критические зоны системы

* платежи
* заказы
* объявления
* поиск
* авторизация
* событийный конвейер (Kafka -> ClickHouse)

---

==== Общий критерий успешности тестирования

Система считается корректной, если:

* 100% критических сценариев проходят успешно
* потери транзакций: 0
* потери событий: 0 (с учетом повторной обработки)
* соблюдение SLA:
- latency ≤ 2 сек (P95)
- search ≤ 5 сек (P95)

* система выдерживает:
- ≥ 10 000 concurrent users (стабильно)
- ≥ 100 000 событий/сек

* восстановление:
- сервисы ≤ 15 мин
- данные ≤ 1 час (OLTP), ≤ 6 часов (OLAP)

* геораспределение не приводит к превышению допустимого уровня рассогласования
